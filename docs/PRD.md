# PRD：中文版面试时间管理系统

## 产品定位

interview-scheduler-cn 是面向面试场景的隐私隔离型时间协调 Web 应用，不是普通投票工具。候选人只提交和查看自己的信息；管理员统一查看候选人可用时间、审核修改、安排面试并锁定已预约时间。

## 角色

- 超级管理员：可以创建多个面试组，访问和管理所有面试组。
- 组级管理员：可以登录后台，但只访问被授权的面试组和由这些面试组派生出的招聘项目。
- 候选人：通过姓名、邮箱、复杂随机组编号请求一次性链接，再以候选人 session 进入候选人端，没有后台权限。

## P0 功能

- 超级管理员可以创建多个面试组。
- 每个面试组由系统生成复杂随机组编号，不允许自增 ID、短码、组名派生码或弱编号。
- 普通管理员可以进入后台，但所有服务端动作必须按面试组角色校验权限。
- 管理员创建面试组时，系统同步创建招聘项目和默认轮次。
- 后台可以按招聘项目查看轮次、关联面试组和项目级面试官池。
- 面试组是候选人数据、预约、审核、统计和备注的安全边界；项目不授予跨组候选人数据访问权。项目内授权组的 `OWNER`/`SCHEDULER` 仅共享面试官池这一排期资源。
- 候选人通过姓名、邮箱、复杂随机组编号请求一次性邮箱访问链接，再通过候选人 session 进入。
- 候选人可填写备注；候选人备注只对管理员和本人有效视图可见，对其他候选人不可见。
- 候选人首次提交直接生效，生成 ACTIVE submission，并设置 candidate.activeSubmissionId。
- 候选人后续修改必须提交审核请求，不立即生效。
- 修改提交前和提交时都要显示清晰提醒：修改需要管理员审核，审核通过后才会生效，审核通过前仍以旧版本为准。
- 同一候选人同一时间最多一个 PENDING_REVIEW submission。
- 管理员可以审核通过或拒绝修改。
- 管理员通过修改前必须再次校验新版本 slot 是否仍开放且未锁定。
- 管理员可以安排面试。
- 管理员安排或调整面试时可以选择项目面试官。
- 同一面试官不能被安排到时间重叠的已确认面试。
- 已预约时间自动锁定，其他候选人端只能看到不可选，不能看到锁定原因。
- 取消预约时释放锁。
- 候选人无法看到其他候选人。
- 候选人无法知道时间不可选原因。
- 管理员可以给每个候选人写管理员私有备注。
- 管理员私有备注只在管理员后台和管理员 API 可见；候选人页面、候选人 API、候选人响应 payload 绝对不能包含该字段。
- 候选人访问链接必须过期且一次性消费，候选人页面不通过 URL query 承载姓名或邮箱。
- 管理员必须邮箱 + 密码登录。
- 管理员密码必须 hash 存储，session 使用 httpOnly cookie。
- 生产环境必须使用可信 HTTPS；HTTP 只允许跳转到 HTTPS，管理员和候选人 cookie 必须带 `Secure` 且限定在应用 basePath。
- 候选人访问链接的 GET 不消费 token；只有确认 POST 可以原子消费并创建 session。

## 关键页面

- 候选人：`/join`、`/candidate/[groupCode]`、已提交页、修改申请页、面试安排页。
- 管理员：`/admin/login`、`/admin`、`/admin/projects`、项目详情、组设置、时间段配置、候选人列表、候选人详情、审核列表/详情、时间总览、预约列表。

## 非目标

- P0 初版不做候选人邮箱验证；当前实现已升级为一次性邮箱访问链接。
- P0 不依赖第三方 OAuth。
- P0 不向候选人展示其他候选人的数量、身份、备注或可用时间。
- P0 不用前端隐藏替代服务端权限校验。

## 成功标准

- `scripts/check.sh` 通过后才允许标记功能完成。
- 候选人 DTO 测试必须证明不包含 `adminNotes`、`internalNote`、`reasonInternal` 等敏感字段。
- 管理员认证、组权限、修改审核、预约锁定都有单测或集成测试覆盖。
- 跨组读写隔离、一次性链接并发消费、DST 缺失/重复本地时间、排期并发冲突和 basePath 重定向都必须有自动化覆盖。
